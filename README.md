# Enron Spam Classifier

A portfolio-ready machine learning project that rebuilds an end-to-end **spam detection pipeline** on the Enron email dataset using **count-based feature engineering** and a **multinomial NaГЇve Bayes classifier**.

This repository is designed to do two things well:

1. **Reproduce the assignment pipeline**  
   It generates the exact deliverables required by the homework:
   - `train-features.tsv`
   - `test-features.tsv`
   - `feature-probabilities.tsv`
   - `train-predictions.tsv`
   - `test-predictions.tsv`
   - `roc.tsv`

2. **Read like a real project**  
   The code is modular, documented, and easy to explain in an interview.

---

## Project overview

The raw dataset contains these columns:

- `Message ID`
- `Subject`
- `Message`
- `Spam/Ham`
- `Date`

The pipeline:

1. loads and validates the raw CSV
2. builds sparse count-based features from **Subject + Message**
3. splits data using the required rule: `Message ID % 30 == 0` for test, otherwise train
4. estimates conditional feature probabilities with **Laplace smoothing** (`alpha = 1`)
5. predicts ham/spam probabilities with a **multinomial NaГЇve Bayes classifier**
6. computes ROC points from thresholds `0.01` to `0.99`
7. saves all outputs as TSV files

---

## Feature engineering

The feature extractor is intentionally simple, interpretable, and appropriate for a multinomial model.

### Features used

- **unigram counts** from the combined subject and message text
- **subject-emphasis tokens** such as `SUBJ_offer` so subject words can carry extra signal
- **structural count features**
  - `HAS_URL`
  - `NUM_EXCLAM`
  - `NUM_DOLLAR`
  - `NUM_DIGIT_TOKENS`

All feature values are stored as integer counts in a sparse JSON dictionary.

Example:

```json
{
  "free": 3,
  "money": 1,
  "SUBJ_offer": 1,
  "HAS_URL": 2,
  "NUM_EXCLAM": 4
}
```

## Results on the provided CSV

Using the included implementation and the uploaded dataset:

- Dataset rows: **33,716**
- Train rows: **32,592**
- Test rows: **1,124**
- Vocabulary size: **172,489**
- Accuracy @ 0.50 threshold: **0.9858**
- Precision @ 0.50 threshold: **0.9810**
- Recall @ 0.50 threshold: **0.9913**
- F1 @ 0.50 threshold: **0.9861**
- ROC AUC: **0.9950**

The ROC curve generated by the pipeline is saved to:

`outputs/roc_curve.png`

## Output files

### `train-features.tsv` and `test-features.tsv`
Two columns:

- `Message ID`
- `features_json`

`features_json` contains a sparse JSON dictionary of feature counts for each email.

### `feature-probabilities.tsv`
Three columns:

- `feature`
- `ham_probability`
- `spam_probability`

These are the conditional probabilities estimated from the training set using additive smoothing with `alpha = 1`.

### `train-predictions.tsv` and `test-predictions.tsv`
Three columns:

- `Message ID`
- `ham`
- `spam`

These are posterior class probabilities from the NaГЇve Bayes classifier.

### `roc.tsv`
Three columns:

- `threshold`
- `false_positive_rate`
- `true_positive_rate`

Thresholds range from `0.01` to `0.99` in increments of `0.01`.

## Design choices

### Why NaГЇve Bayes?
NaГЇve Bayes is a strong baseline for text classification because it is:

- fast
- interpretable
- naturally compatible with sparse count features
- easy to explain mathematically

### Why sparse JSON features?
Email text creates a very large vocabulary. Sparse storage keeps the output compact and matches how text models are typically represented.

### Why log probabilities?
Multiplying many small conditional probabilities causes numerical underflow. The classifier therefore works in **log space** and converts back to normalized probabilities at the end.
